Software Development Life Cycle (SDLC) Models

Water Fall Model = linear-sequential life cycle model.

  • Adv - simple ,understand, easy to manage, one at a time.
  • Disadv - difficult to go back and change, No working until late, High risk, Not a good, long and ongoing projects.
  • When to use - requirements are very well known, clear and fixed, short project

V Model = sequential manner in V-shape, Each phase completed before next phase begins. Testing parallel

  • Adva - Simple and easy to use, Testing before coding, saves a lot of time, Proactive defect tracking
  • Disadv - Software is developed on implementation phase, so no early prototypes , If any changes in midway, documents to be updated.
  • When to use - small to medium sized projects, requirements are clearly defined and fixed, ample technical resources with needed technical expertise.

Prototype Model = attractive duplicated and idea requirements.

  • Adva = actively involved in dev, better understanding, Errors detected earlier, Quicker user feedback., Missing functionality identified, Confusing or difficult functions identified
  • Disadv = scope may expand beyond original plans, Incomplete application not to be used as full system was designed
  • When to use = desired system needs have a lot of interaction with end users.

Agile Model = iterative and incremental approach, rapids mall incremental cycles releases.

  • Adva - Good communication b/w team, Working software is delivered frequently, no need to wait, Continuous improve tech & design, Even late/middle changes in requirements are welcomed.
  • Disadv - effort required at the beginning, project easily get taken off track if is not clear what final outcome that they want, Only senior programmers are capable of taking decisions, 
  • When to use - changes at very little cost, new feature work of a few days, limited planning is required to get started.

Test Metrics

Generation of Software Test Metrics is the most important responsibility of the Software Test Lead/Manager.

 

Effectiveness: Doing the right thing. It deals with meeting the desirable attributes that are expected by the customer.

Efficiency: Doing the thing right. It concerns the resources used for the service to be rendered


a. Test Plan coverage on Functionality:

(No of req covered / total # of req) * 100

b. Test Case defect density:

(Defective # Test Scripts /Total # Test req Scripts) * 100

c. Defect Slippage/rejection Ratio:

# of Defects Slipped / (# of Defects Raised - # of Defects Withdrawn)

d. Requirement Volatility:

(# of req Added + Deleted + Modified) *100 / # of Original req

e. Review Efficiency:

100*Total # of defects found by reviews/Total # of project defects

f. Defect Removal Effectiveness(DRE):

DRE= (Defects removed during development phase x100%) / Defects latent in the product


Defects latent in the product = Defects removed during development Phase+ defects found later by user
